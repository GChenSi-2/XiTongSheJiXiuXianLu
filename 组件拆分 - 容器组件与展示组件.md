# 组件拆分：容器组件与展示组件

## 起源与背景

这一模式由 Dan Abramov 在 2015 年的博客文章 *Presentational and Container Components* 中正式提出，是 React 生态早期最广为流传的架构思想之一。

其核心动机来自**单一职责原则（SRP）**：一个组件不应该同时负责「从哪里取数据」和「如何显示数据」这两件事。将二者分离，可以让每部分都更小、更纯粹、更容易测试和复用。

> [!note] Hooks 时代的演变
> 2019 年 Dan Abramov 在原文末尾补充说：*"I no longer suggest splitting your components like this."*  
> 原因是 **Custom Hooks** 出现后，逻辑与 UI 的分离可以通过 `useSomething()` 实现，不再强制要求组件层的二分法。  
> 但**关注点分离的思想本身**依然成立，深刻理解这一模式仍是写好组件的基础。

---

## 核心定义

### 展示组件（Presentational Component）

> **只关心「看起来是什么样子」**

| 特征              | 说明                                 |
| --------------- | ---------------------------------- |
| 纯 UI            | 只负责渲染，不关心数据来自哪里                    |
| 通过 `props` 接收一切 | 数据、回调函数全部由外层传入                     |
| 无副作用            | 不做 `fetch`、不写 `localStorage`、不订阅事件 |
| 无业务 state       | 最多只有 UI 状态（如 `isHovered`、`isOpen`） |
| 可复用             | 换一个数据源就能用在另一个地方                    |
| 易于测试            | 给定同样的 props，输出永远相同                 |

```tsx
// ✅ 纯展示组件：只负责渲染，数据全部来自 props
interface UserCardProps {
  name: string;
  avatarUrl: string;
  bio: string;
  onFollow: () => void;
}

function UserCard({ name, avatarUrl, bio, onFollow }: UserCardProps) {
  return (
    <div className="user-card">
      <img src={avatarUrl} alt={name} />
      <h3>{name}</h3>
      <p>{bio}</p>
      <button onClick={onFollow}>关注</button>
    </div>
  );
}
```

---

### 容器组件（Container Component）

> **只关心「做什么事情」**

| 特征 | 说明 |
|------|------|
| 负责数据获取 | `fetch`、订阅 Store、调用 API |
| 管理业务状态 | `useState`、`useReducer`、从 Context 读取 |
| 处理副作用 | `useEffect` 发请求、订阅、定时器 |
| 不写样式 | 不关心 CSS，几乎不带 HTML 结构 |
| 组装展示组件 | 把数据和回调传给子级展示组件 |

```tsx
// ✅ 容器组件：负责数据和逻辑，几乎不写 UI
function UserCardContainer({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetchUser(userId).then(data => {
      setUser(data);
      setLoading(false);
    });
  }, [userId]);

  const handleFollow = useCallback(async () => {
    await followUser(userId);
  }, [userId]);

  if (loading) return <Spinner />;
  if (!user) return <ErrorMessage />;

  // 把数据"喂给"展示组件
  return (
    <UserCard
      name={user.name}
      avatarUrl={user.avatarUrl}
      bio={user.bio}
      onFollow={handleFollow}
    />
  );
}
```

## 如何判断一个组件属于哪类

用以下问题做快速诊断：

```
这个组件是否直接调用了 fetch / 读取了 Redux Store / 访问了 Context？
├─ 是 → 容器组件（或至少包含容器逻辑）
└─ 否 → 进一步判断
      这个组件的所有数据都来自 props 吗？
      ├─ 是 → 展示组件
      └─ 否（有自己的业务 state）→ 容器组件
```

> [!tip] 混合型组件也可以接受
> 实践中不必强行二分。一个组件可以同时有少量 UI state 和少量逻辑，只要职责不过于复杂就可以。重要的是**保持清晰的职责边界**，而非套用死板的分类标签。

## 应用实例

### 实例 1：Tooltip 组件

参见 [[Tooltip 组件拆分设计]] 中的完整实现，是这一模式的典型应用：

| 组件             | 类型  | 职责                                         |
| -------------- | --- | ------------------------------------------ |
| `QuestionIcon` | 展示  | 画问号圆圈，不知道自己会被放在哪里                          |
| `TooltipBox`   | 展示  | 画深色文字框，不知道定位                               |
| `Tooltip`      | 容器  | 管理 `visible` state、计算坐标、绑定鼠标事件、用 Portal 渲染 |

关键收益：
- 更换图标只改 `QuestionIcon`，定位逻辑完全不动
- 更换文字框样式只改 `TooltipBox`，交互逻辑完全不动
- 展示组件可以在 Storybook 中单独预览，无需模拟任何交互

### 实例 2：评论列表

```tsx
/* ========== 展示组件 ========== */

// 单条评论——纯 UI
function CommentItem({ author, content, createdAt, onDelete }: CommentItemProps) {
  return (
    <li>
      <strong>{author}</strong>
      <span>{createdAt}</span>
      <p>{content}</p>
      <button onClick={onDelete}>删除</button>
    </li>
  );
}

// 整个列表——纯 UI，数据由外部传入
function CommentList({ comments, onDelete }: CommentListProps) {
  if (comments.length === 0) return <p>暂无评论</p>;
  return (
    <ul>
      {comments.map(c => (
        <CommentItem key={c.id} {...c} onDelete={() => onDelete(c.id)} />
      ))}
    </ul>
  );
}

/* ========== 容器组件 ========== */

function CommentListContainer({ postId }: { postId: string }) {
  const [comments, setComments] = useState<Comment[]>([]);

  useEffect(() => {
    getComments(postId).then(setComments);
  }, [postId]);

  const handleDelete = useCallback(async (commentId: string) => {
    await deleteComment(commentId);
    setComments(prev => prev.filter(c => c.id !== commentId));
  }, []);

  return <CommentList comments={comments} onDelete={handleDelete} />;
}
```

**收益**：
- `CommentList` 和 `CommentItem` 可以用假数据直接预览，不依赖 API
- 切换数据源（如从 REST 换成 WebSocket）只改容器，展示层零改动

### 实例 3：Hooks 时代的等价写法

Hooks 出现后，容器逻辑可以提取到 Custom Hook，组件树本身变得扁平：

```tsx
// 把容器逻辑提到 hook
function useComments(postId: string) {
  const [comments, setComments] = useState<Comment[]>([]);

  useEffect(() => {
    getComments(postId).then(setComments);
  }, [postId]);

  const deleteComment = useCallback(async (id: string) => {
    await deleteCommentApi(id);
    setComments(prev => prev.filter(c => c.id !== id));
  }, []);

  return { comments, deleteComment };
}

// 组件本身只负责渲染，逻辑来自 hook
function CommentSection({ postId }: { postId: string }) {
  const { comments, deleteComment } = useComments(postId);

  return <CommentList comments={comments} onDelete={deleteComment} />;
}
```

这和「容器 + 展示」模式在本质上是**同一种思路**：把**数据逻辑**和**渲染逻辑**物理隔离。区别只是隔离的边界从「组件层」移到了「Hook 层」。

## 最佳实践

### ✅ 1. 展示组件的 props 只接受「数据形状」，不接受原始 Model

```tsx
// ❌ 展示组件依赖了 User model，换数据结构就要改 UI
function UserCard({ user }: { user: User }) {
  return <div>{user.profile.displayName}</div>;
}

// ✅ 展示组件只需要它真正需要渲染的字段
function UserCard({ name, avatarUrl }: { name: string; avatarUrl: string }) {
  return <div>{name}</div>;
}
```

---

### ✅ 2. 容器组件不写样式，展示组件不做 fetch

这是两类组件最重要的**硬边界**。一旦容器组件开始关心颜色、字号，或者展示组件开始 `fetch`，拆分的价值就打了折扣。

---

### ✅ 3. 展示组件尽量写成函数组件 + 无副作用

没有 `useEffect`、没有 `useRef` 操作 DOM（除非是纯 UI 动画需求）的展示组件，天然就是「纯组件」：

```tsx
// 给定同样的 props → 永远输出同样的 JSX
// React.memo 优化毫无副作用，可以放心使用
const StatusBadge = React.memo(function StatusBadge({ status }: { status: 'active' | 'inactive' }) {
  const color = status === 'active' ? '#22c55e' : '#9ca3af';
  return <span style={{ color }}>● {status}</span>;
});
```

---

### ✅ 4. 让 Storybook / 单元测试驱动展示组件设计

展示组件的最大价值之一就是可以独立开发和测试。如果写了一个展示组件却无法脱离容器单独渲染，说明 props 设计有问题：可能还耦合了某个全局状态或副作用。

---

### ✅ 5. 一个容器只对应一个核心业务单元

避免写一个「万能容器」同时获取用户信息、评论列表、推荐商品。粒度过大的容器和没有拆分一样难维护。

---

## 常见误区

> [!warning] 误区 1：把所有状态都赶进容器
> 纯粹的 UI 状态（如 hover、展开/收起的动画）**不需要**提升到容器。展示组件可以用 `useState` 管理自己的视觉状态。

> [!warning] 误区 2：容器组件一定要是「页面级」的
> 容器/展示是**职责上的分层**，不是规模上的分层。一个小小的 `Tooltip` 也可以有容器层（管理 `visible` 和坐标计算）。

> [!warning] 误区 3：用了 Redux / Zustand 就不需要这个模式
> 状态管理库只是把状态从本地移到了 Store，**读取 Store 的那一层仍然是容器逻辑**。`useSelector` 出现在哪个组件里，哪个组件就承担了容器职责。

> [!warning] 误区 4：展示组件绝对不能有任何 state
> 展示组件可以有 UI state（动画、本地 hover 效果），但**不能有业务 state**（用户数据、服务端状态）。

## 模式对比总览

| 维度   | 展示组件             | 容器组件                   |
| ---- | ---------------- | ---------------------- |
| 关注点  | 外观（How it looks） | 行为（How it works）       |
| 数据来源 | `props`          | `fetch`、Store、Context  |
| 副作用  | ❌ 无              | ✅ `useEffect`          |
| 样式   | ✅ 有              | ❌ 几乎无                  |
| 可复用性 | ✅ 高              | ⚠️ 低（与业务耦合）            |
| 测试难度 | ✅ 容易（纯函数式）       | ⚠️ 需要 mock             |
| 类比   | HTML 模板 / View   | Controller / ViewModel |

## 延伸学习资源

### 书籍

| 书名 | 作者 | 推荐理由 |
|------|------|----------|
| *Advanced React*（2023） | Nadia Makarevich | **最直接对口**。全书核心就是「如何写出可复用、可组合的组件」，覆盖 Compound Component、Render Props、Context 设计、性能优化等。作者为前 Atlassian 高级工程师，例子全来自真实工程场景。官网：[advanced-react.com](https://advanced-react.com) |
| *Fluent React*（O'Reilly 2024） | Tejas Kumar | 偏向理解 React 工作原理（reconciler、fiber），从底层理解为什么要这样拆组件。适合已有一定基础、想知道「为什么」的阶段。 |

### 在线课程

| 课程 | 平台 | 推荐理由 |
|------|------|----------|
| *Epic React* | [epicreact.dev](https://epicreact.dev) | React 社区公认最深入的进阶课程，以 Workshop 代码练习形式覆盖 Custom Hooks 设计、Compound Component、State Reducer、Control Props 等高级模式。Kent C. Dodds 本人也是容器/展示模式在 Hooks 时代演进的重要推动者。 |
| *The Joy of React* | [joyofreact.com](https://www.joyofreact.com) | 风格友好，适合打磨「组件思维」的直觉。 |

**[patterns.dev/react](https://www.patterns.dev/react)**：专门讲 JavaScript / React 设计模式，每个模式都有交互式代码示例，直接覆盖 Presentational/Container、HOC、Render Props、Hooks Pattern、Compound Pattern。

```
你现在的位置（已懂容器/展示、Hooks）
        ↓
patterns.dev  ← 免费，快速浏览 Compound / Render Props 等进阶模式（1–2 天）
        ↓
Advanced React（书）← 系统补全「组件 API 设计」的完整知识体系
        ↓
Epic React（课）← 用练习题把模式真正内化
```

> [!tip] 判断一本书/课程是否真的在讲组件设计
> 如果内容里出现以下关键词，说明它确实覆盖了组件拆分的进阶技巧：
> - **Compound Component**（复合组件）
> - **Render Props**
> - **Inversion of Control**（控制反转）
> - **Component Composition**
>
> 容器/展示模式是这个大主题的**入门款**，上面这几个模式是它的进阶延伸。
